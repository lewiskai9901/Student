## 一、核心问题识别

### 1.1 当前设计的根本问题

| 问题 | 说明 |
|------|------|
| 导出中心独立化 | 错误地将导出功能独立出来，但实际上每个检查计划应有自己独立的导出方案 |
| 导出方案不可自定义 | 缺少Word模板编辑器，无法自定义标题、正文、表格列、落款等 |
| 轮次硬编码 | 轮次数量固定为1/2/3，但实际每次检查的轮次是动态的 |
| 功能分散 | 统计、评级、通报功能分散在不同页面，应该整合到检查计划中 |

### 1.2 核心设计原则

```
┌─────────────────────────────────────────────────────────────┐
│                        检查计划                              │
│  ┌─────────────────────────────────────────────────────────┐│
│  │ 一个检查计划 = 一套完整的检查、通报、评级、统计体系      ││
│  │                                                         ││
│  │  • 使用同一个检查模板（检查结构固定）                    ││
│  │  • 拥有独立的导出模板配置                                ││
│  │  • 拥有独立的评级规则                                    ││
│  │  • 拥有独立的统计维度                                    ││
│  └─────────────────────────────────────────────────────────┘│
│                                                             │
│  变动因素：                                                  │
│    1. 每次检查的轮次数量（动态）                             │
│    2. 检查目标（个别目标可能不参与）                         │
└─────────────────────────────────────────────────────────────┘
```

## 二、三大核心功能深度分析

### 2.1 扣分通报（学生通报）

#### 使用场景

**场景1：单轮实时通报**
```
早晨7:30检查迟到 → 录入违纪学生 → 第一轮结束
                                    ↓
                        立即导出通报（检查尚未完全结束）
                                    ↓
                        Word/PDF形式发布
```

**场景2：多轮汇总通报**
```
早（第1轮） + 中（第2轮） + 晚（第3轮） 检查完成
                    ↓
        选择多个轮次，汇总导出通报
```

**场景3：多检查汇总通报**
```
周一检查 + 周三检查 + 周五检查
                ↓
    选择多个检查记录，汇总导出通报
    （前提：使用同一检查模板）
```

### 2.3 统计分析

#### 统计维度

| 维度 | 说明 |
|------|------|
| 按天 | 每天的扣分情况 |

## 三、系统架构设计

### 3.1 整合到检查计划

```
检查计划详情页
├── 📋 基本信息
├── 📝 检查记录
│     └── 检查列表（每条可展开查看轮次）
├── 📢 通报管理  ← 新模块
│     ├── 通报模板配置
│     └── 生成通报
├── 🏆 评级管理  ← 新模块
│     ├── 评级规则配置
│     ├── 执行评级
│     └── 评级统计
└── 📊 统计分析  ← 新模块
      ├── 统计报表
      └── 导出报表
```

### 3.2 数据模型

```
CheckPlan (检查计划)
│
├── CheckTemplate (检查模板) - 1:1
│     ├── Category (检查类别) - 1:N
│     └── DeductionItem (扣分项) - 1:N
│
├── CheckRecord (检查记录) - 1:N
│     ├── CheckRound (检查轮次) - 1:N （动态）
│     │     └── roundNumber, roundName, startTime, endTime
│     └── Deduction (扣分明细) - 1:N
│           └── studentId, itemId, score, roundId
│
├── NoticeTemplate (通报模板) - 1:N
│     ├── templateName
│     ├── headerContent (标题区)
│     ├── bodyStartContent (正文开头)
│     ├── columnConfig (表格列配置 JSON)
│     ├── bodyEndContent (正文结尾)
│     └── footerContent (落款区)
│
├── RatingRule (评级规则) - 1:N
│     ├── ruleName (评级名称)
│     ├── scoreSource (total / categoryId)
│     ├── ratingMethod (top_n / top_percent / threshold)
│     ├── ratingValue (具体数值)
│     └── levelConfig (等级配置 JSON)
│
├── RatingResult (评级结果)
│     ├── ruleId
│     ├── checkRecordIds (可能多个)
│     ├── classId
│     ├── levelName
│     ├── score
│     └── createdAt
│
└── NoticeRecord (通报记录)
      ├── templateId
      ├── checkRecordIds
      ├── roundIds
      ├── generatedAt
      └── filePath
```

### 3.3 技术方案

#### Word模板编辑器

**方案选择：富文本编辑器 + 变量占位符**

```
使用 TinyMCE / Quill / WangEditor 作为富文本编辑器
                    ↓
支持插入变量占位符，如：
  {{检查名称}}  {{检查日期}}  {{班级名称}}
                    ↓
表格区域使用特殊标记：
  {{#TABLE}}...{{/TABLE}}
                    ↓
后端使用模板引擎（如 FreeMarker / Thymeleaf）渲染
                    ↓
使用 Apache POI 或 docx4j 生成 Word
使用 iText 或 OpenPDF 生成 PDF
```

#### 动态轮次

```
每次检查开始时，可设置轮次：
  - 输入轮次数量
  - 为每个轮次命名（如：早检、午检、晚检）

或者动态添加：
  - 开始时只有1轮
  - 需要时可"添加新轮次"
```

**问题：模板编辑器需要支持到什么程度？**

- [ ] 选项B：富文本编辑（字体、颜色、对齐等可调）
- [ ] 选项C：支持上传已有Word模板，系统识别占位符

### 4.2 评级触发时机

**问题：评级是自动触发还是手动触发？**

- [ ] 选项A：每次检查完成后自动评级
- [ ] 选项B：用户手动选择检查记录，发起评级
- [ ] 选项C：两者都支持（可配置）

### 4.3 评级结果是否可修改

**问题：评级结果生成后是否可以撤销或修改？**

- [ ] 选项A：可以撤销/删除
- [ ] 选项B：可以修改
- [ ] 选项C：不可更改（作为历史记录）

### 4.4 多检查汇总规则

**问题：汇总时如何处理同一学生的多次违纪？**

- [ ] 选项A：每次违纪单独列出
- [ ] 选项B：合并为一行，分数累加
- [ ] 选项C：用户可选择

### 4.5 表格列的自定义范围

**问题：表格列是从预定义列表选择，还是支持完全自定义？**

- [ ] 选项A：从预定义字段中选择（序号、姓名、班级、扣分项等）
- [ ] 选项B：支持添加自定义字段
- [ ] 选项C：支持计算字段（如：违纪次数统计）

### 4.6 检查计划的生命周期

**问题：一个检查计划的使用周期是多长？**

- [ ] 选项A：学期级（如"2024年下学期卫生检查"）
- [ ] 选项B：学年级
- [ ] 选项C：长期持续

**子问题：计划结束后，历史数据如何处理？**

### 4.7 轮次命名规则

**问题：轮次名称是用户自定义还是系统预设？**

- [ ] 选项A：用户自定义（如：早检、午检、晚检）
- [ ] 选项B：系统预设（第1轮、第2轮...）

## 七、技术难点分析

### 7.1 Word模板编辑器

**难点：**
- 富文本编辑器的选择和集成
- 变量占位符的插入和识别
- 表格区域的特殊处理
- 预览功能
- 后端模板渲染

**建议方案：**
使用 WangEditor（国产，中文友好）+ 自定义变量插件

### 7.2 动态轮次管理

**难点：**
- 轮次数量不固定
- 支持实时添加轮次
- 轮次状态管理（进行中/已完成）
- 与扣分记录的关联

### 7.3 多检查汇总

**难点：**
- 确保被汇总的检查使用同一模板
- 数据合并逻辑
- 去重规则
- 分数计算规则

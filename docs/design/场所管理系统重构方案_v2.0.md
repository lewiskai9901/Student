## 一、现状分析

### 1.1 当前架构

```
places 表（场所基础信息）
├─ org_unit_id              # 所属组织（单一字段）
├─ responsible_user_id      # 负责人
└─ 其他字段...

place_org_relations 表（场所-组织关系）
├─ place_id
├─ org_unit_id
├─ relation_type (PRIMARY/SHARED/MANAGED)
├─ can_use / can_manage / can_inspect
└─ 权限控制字段...

place_occupants 表（入住管理）
├─ place_id
├─ occupant_id
├─ position_no             # 位置号
└─ 入住/退出时间...
```

### 1.2 当前问题

1. **语义不清**：`org_unit_id` 和 `place_org_relations` 的区分不明确
   - 用户困惑：一个场所到底"属于"谁？
   - 继承逻辑缺失：子场所是否继承父场所的组织？

2. **功能缺失**：
   - 无使用历史追踪
   - 无容量变更记录
   - 无场所状态变更审计
   - 无批量操作支持

3. **查询效率**：
   - 查询"某组织的所有场所"需JOIN多表
   - 树形结构查询无索引优化

## 三、重构方案设计

### 3.1 核心设计原则

根据 DDD（领域驱动设计）和业界实践，确定以下原则：

| 原则 | 说明 | 实现 |
|------|------|------|
| **单一所有权** | 每个场所有且仅有一个管理责任方 | `org_unit_id` - 表示管理归属 |
| **使用权分离** | 所有权 ≠ 使用权 | `place_org_relations` - 表示使用权限 |
| **显式继承** | 继承关系必须明确，存储到数据库 | 子场所创建时自动继承，但显式存储 |
| **审计追踪** | 关键变更可追溯 | 新增 `place_change_history` 表 |
| **查询优化** | 支持高效的树形查询 | 物化路径 + 索引优化 |

---

### 3.2 字段语义重新定义

#### **places 表字段**

| 字段 | 新语义 | 示例 | 说明 |
|------|--------|------|------|
| `org_unit_id` | **管理责任方**<br>谁负责维护、保洁、设施管理 | 宿舍楼 → 后勤处<br>实验室 → 资产处 | **存储继承**：创建子场所时，如未设置则自动填充父场所的值 |
| `responsible_user_id` | 具体负责人 | 张三（后勤处员工） | 可空，表示具体责任到人 |
| `tree_path` | 树形路径（已有） | `/1/5/12` | 用于树形查询优化 |
| `level` | 层级深度（已有） | 3 | 根节点为1 |
| `status` | 场所状态 | 0停用/1正常/2维护中 | 生命周期管理 |

#### **place_org_relations 表（使用关系）**

| 字段 | 语义 | 示例 |
|------|------|------|
| `relation_type` | 关系类型 | PRIMARY（主要使用方）<br>SHARED（共用）<br>MANAGED（托管） |
| `can_use` | 可使用 | 学生可入住宿舍 |
| `can_manage` | 可管理 | 院系可管理本院宿舍 |
| `can_inspect` | 可检查 | 学工处可检查所有宿舍 |
| `can_assign` | 可分配 | 教务处可分配教室 |
| `start_date` | 开始日期 | 2024-09-01 |
| `end_date` | 结束日期 | 2025-06-30 |

### 3.4 继承策略实现

#### **方案：存储继承值 + 界面标识**

```java
// 后端逻辑（CreatePlace）
public Long createPlace(CreatePlaceCommand command) {
    // 1. 获取父场所
    UniversalPlace parent = null;
    if (command.getParentId() != null) {
        parent = placeRepository.findById(command.getParentId())
                .orElseThrow(() -> new IllegalArgumentException("父场所不存在"));
    }

    // 2. 继承所属组织
    Long orgUnitId = command.getOrgUnitId();
    if (orgUnitId == null && parent != null) {
        orgUnitId = parent.getOrgUnitId();  // 自动继承
    }

    // 3. 创建场所
    UniversalPlace place = UniversalPlace.create(...);
    place.setOrgUnitId(orgUnitId);  // 存储继承值

    // 4. 记录继承信息（用于前端显示）
    boolean isInherited = (command.getOrgUnitId() == null && parent != null);

    return placeRepository.save(place).getId();
}
```

#### **前端显示逻辑**

```typescript
// PlaceTreeNode DTO 新增字段
interface PlaceTreeNode {
  orgUnitId: number
  orgUnitName: string
  parentOrgUnitId?: number  // 新增：父场所的组织ID
  isOrgInherited: boolean   // 新增：是否继承
}

// 前端显示
<span v-if="node.isOrgInherited" class="text-gray-400 text-[10px]">(继承)</span>
```

#### **H. 物化视图优化**

**参考**：Google Cloud BigQuery Materialized Views

**场景**：
```sql
-- 问题：查询"某组织可使用的所有场所"需多表JOIN
-- 方案：创建物化视图

CREATE MATERIALIZED VIEW mv_org_accessible_places AS
SELECT
  o.org_unit_id,
  p.place_id,
  p.place_name,
  r.relation_type,
  r.can_use,
  r.can_manage,
  r.can_inspect
FROM place_org_relations r
JOIN places p ON r.place_id = p.id
JOIN org_units o ON r.org_unit_id = o.id
WHERE r.deleted = 0 AND p.deleted = 0;

-- 查询速度提升10-100倍
```

```sql
-- 变更历史表
CREATE TABLE place_change_history (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  place_id BIGINT NOT NULL COMMENT '场所ID',
  change_type VARCHAR(50) NOT NULL COMMENT '变更类型：ORG_TRANSFER/CAPACITY_CHANGE/STATUS_CHANGE/RESPONSIBLE_CHANGE',
  field_name VARCHAR(100) COMMENT '字段名',
  old_value TEXT COMMENT '旧值',
  new_value TEXT COMMENT '新值',
  changed_by BIGINT COMMENT '操作人ID',
  changed_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '变更时间',
  reason VARCHAR(500) COMMENT '变更原因',
  ip_address VARCHAR(50) COMMENT '操作IP',
  INDEX idx_place_time (place_id, changed_at DESC),
  INDEX idx_change_type (change_type, changed_at DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='场所变更历史';

-- 标签表（可选）
CREATE TABLE place_tags (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  place_id BIGINT NOT NULL COMMENT '场所ID',
  tag_key VARCHAR(100) NOT NULL COMMENT '标签键',
  tag_value VARCHAR(200) NOT NULL COMMENT '标签值',
  created_by BIGINT COMMENT '创建人',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uk_place_tag (place_id, tag_key),
  INDEX idx_tag_key_value (tag_key, tag_value)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='场所标签';

-- 预订表（可选）
CREATE TABLE place_bookings (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  place_id BIGINT NOT NULL,
  org_unit_id BIGINT COMMENT '预订组织',
  user_id BIGINT NOT NULL COMMENT '预订人',
  start_time DATETIME NOT NULL,
  end_time DATETIME NOT NULL,
  status VARCHAR(20) NOT NULL DEFAULT 'PENDING' COMMENT 'PENDING/APPROVED/REJECTED/CANCELLED',
  purpose VARCHAR(500) COMMENT '用途',
  approved_by BIGINT COMMENT '审批人',
  approved_at DATETIME COMMENT '审批时间',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  deleted TINYINT(1) DEFAULT 0,
  INDEX idx_place_time (place_id, start_time, end_time),
  INDEX idx_user (user_id, created_at DESC),
  INDEX idx_status (status, start_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='场所预订';
```

## 九、性能优化

### 9.1 查询优化

```sql
-- 场景：查询某组织的所有可用场所（直接归属 + 使用关系）
-- 优化前：3次查询 + 2次JOIN
-- 优化后：1个复杂查询

SELECT DISTINCT
  p.id, p.place_name, p.org_unit_id,
  CASE
    WHEN p.org_unit_id = ? THEN 'OWNER'
    ELSE r.relation_type
  END as access_type
FROM places p
LEFT JOIN place_org_relations r ON p.id = r.place_id AND r.org_unit_id = ?
WHERE (p.org_unit_id = ? OR r.org_unit_id IS NOT NULL)
  AND p.deleted = 0
  AND p.status = 1;
```

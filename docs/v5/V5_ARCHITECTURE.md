## 二、重构目标

### 2.1 核心问题

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           重构目标                                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  权限系统优化:                                                               │
│  ├── 功能权限与数据权限彻底分离                                              │
│  ├── 无硬编码：权限、模块、范围全部可配置                                     │
│  ├── 模块级数据权限：同一用户不同模块可有不同数据范围                         │
│  └── 支持复杂场景：跨部门、临时授权、自定义范围                               │
│                                                                             │
│  量化检查系统 V5:                                                            │
│  ├── Session-based 架构：项目→任务→记录三级结构                              │
│  ├── 5种录入模式：SPACE/PERSON/CLASS/ITEM/CHECKLIST                         │
│  ├── 混合宿舍分数分配                                                        │
│  ├── 完整申诉流程                                                            │
│  ├── 整改工单管理                                                            │
│  └── 与权限系统深度整合                                                      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 设计原则

| 原则 | 说明 |
|------|------|
| **部门不分配功能权限** | 业界主流做法，部门只用于组织归属和数据范围 |
| **角色 = 功能权限 + 数据权限** | 角色同时配置两个维度 |
| **无硬编码** | 所有配置项从数据库读取，支持动态扩展 |
| **模块级数据权限** | 同一角色在不同模块可有不同数据范围 |
| **快照模式** | 检查记录保存时固化关联信息 |
| **功能权限与数据权限是AND关系** | 必须同时满足 |

## 五、决策详情

### 决策18：多角色数据权限合并策略

**问题**：用户同时拥有多个角色，且在同一模块有不同数据范围时如何合并？

**决策**：自定义范围和仅本人取**并集**，而非只取最大范围

**合并规则**：
1. 有 ALL → 直接返回 ALL
2. 收集所有范围（并集）
3. 多条件用 OR 连接

**SQL示例**：
```sql
-- 张老师: 班主任(1班) + 检查员(仅本人)
WHERE (
    class_id IN (101)           -- 班主任的自定义范围
    OR created_by = 12345       -- 检查员的仅本人
)
```

---

### 决策19：班主任管理多个班级

**问题**：一个老师同时当多个班的班主任，怎么配置？

**决策**：一个角色配置多个班级范围

```yaml
用户: 张老师
角色: 班主任
数据权限:
  student: CUSTOM
  自定义范围:
    - 类型: CLASS, ID: 101, 名称: 2024级软件1班
    - 类型: CLASS, ID: 102, 名称: 2024级软件2班
```

---

### 决策20：检查记录的部门归属

**问题**：检查记录的 `org_unit_id` 应该填什么？

**决策**：归属到**被检查对象的部门**（快照模式）

**规则**：
- STUDENT → 学生所在班级的部门
- CLASS → 班级的部门
- DORMITORY/CLASSROOM → 场所分配的班级部门

---

### 决策21：学生查看个人扣分记录

**问题**：学生使用"仅本人"范围，但检查记录是检查员创建的

**决策**：学生模块使用专门的过滤字段 `target_id`

```sql
-- 学生个人检查记录模块 (学生专用)
INSERT INTO data_modules (module_code, filter_fields) VALUES
('inspection_personal', '{"student": "target_id"}');
```

---

### 决策22：数据权限缓存失效策略

**问题**：权限变更时如何处理已登录用户的缓存？

**决策**：实时失效，通过 Redis 发布订阅通知所有实例

**好处**：
- 权限变更立即生效，安全性高
- 分布式环境下所有实例同步失效

---

### 决策23：无角色用户的默认行为

**问题**：用户没有分配任何角色时如何处理？

**决策**：可登录但无业务数据访问权限

**行为**：
- 可以登录系统
- 可以查看个人信息、修改密码
- 业务模块菜单不显示
- 访问业务API返回空数据

---

### 决策24：检查记录历史数据迁移

**问题**：现有记录没有快照字段

**决策**：保持空值，查询时兼容处理

```java
// 优先使用快照字段，为空时回退到关联查询
return String.format(
    "(snapshot_department_id IN (%s) OR " +
    "(snapshot_department_id IS NULL AND org_unit_id IN (%s)))",
    idList, idList
);
```

---

### 决策25：API兼容性处理

**问题**：V2 API是否需要与V1兼容？

**决策**：直接替换，前端同步更新

**理由**：
- 前端代码可控，可以同步修改
- 减少维护两套API的成本

---

### 决策26：批量操作的权限校验

**问题**：批量操作时如何处理权限校验？

**决策**：部分成功模式，返回跳过的记录列表

```typescript
ElMessage.warning(
  `成功删除 ${result.successCount} 条，` +
  `${noPermissionCount} 条因权限不足被跳过`
)
```

---

### 决策27：功能权限与数据权限的关系

**问题**：两种权限是什么关系？

**决策**：AND关系，必须同时满足

```
用户请求 → 功能权限检查 → 数据权限过滤 → 返回结果
```

| 功能权限 | 数据权限 | 结果 |
|---------|---------|------|
| ✅ 有 | ALL | 可查看所有 |
| ✅ 有 | DEPARTMENT | 只能看本部门 |
| ❌ 无 | 任意 | 403 禁止访问 |

---

### 决策28：前端路由守卫与数据权限

**问题**：路由守卫是否需要考虑数据权限？

**决策**：路由只检查功能权限，数据权限在页面内处理

**好处**：
- 路由守卫逻辑简单
- 数据权限的空状态可以给用户更友好的提示

## 七、风险与建议

### 7.1 风险点

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 权限配置复杂 | 管理员配置困难 | 提供预设角色模板，简化配置界面 |
| 数据权限性能 | 查询变慢 | 优化SQL，增加索引，考虑缓存 |
| 数据迁移 | 历史数据丢失 | 制定详细迁移计划，保留备份 |
| 并发问题 | 数据不一致 | 使用乐观锁，事务隔离 |

### 7.2 建议

1. **权限缓存**：用户权限缓存到 Redis，角色变更时主动清除
2. **数据权限性能**：大数据量时考虑临时表或 JOIN 优化
3. **模板版本控制**：项目创建时复制模板快照，原模板修改不影响已创建项目
4. **异步处理**：汇总生成、大量数据导出使用后台任务

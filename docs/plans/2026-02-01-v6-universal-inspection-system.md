## 一、系统概述

### 1.1 设计目标

构建一套**通用的量化检查系统**，适用于任何具有"组织-场所-用户"结构的场景：

| 应用场景 | 组织示例 | 场所示例 | 用户示例 | 检查类型 |
|---------|---------|---------|---------|---------|
| 学校 | 系部→班级 | 宿舍、教室 | 学生、教师 | 卫生、纪律、安全 |
| 企业 | 部门→团队 | 办公室、车间 | 员工 | 绩效、合规、安全 |
| 物业 | 小区→楼栋 | 公共区域 | 住户、保安 | 卫生、设施、安全 |
| 医院 | 科室→病区 | 病房、诊室 | 医护人员 | 卫生、安全、服务 |
| 政府 | 局→科室 | 办公室、窗口 | 公务员 | 作风、效能、廉政 |

### 1.2 核心特性

```
┌─────────────────────────────────────────────────────────────────────┐
│                        核心特性                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ✓ 三大检查主体: 组织、场所、用户 - 统一抽象，灵活组合               │
│  ✓ 嵌套检查: 支持多层级递归检查，如 系部→班级→宿舍→床位             │
│  ✓ 混合打分: 同一检查可混合使用扣分制、评分制、评级制等              │
│  ✓ 自定义公式: 用户可定义最终分数计算公式                            │
│  ✓ 智能加权: 按人数/容量加权，公平权重解决检查次数不均问题           │
│  ✓ 动态分组: 支持按属性（年级、专业等）动态创建检查目标分组          │
│  ✓ 响应式UI: PC端双栏布局，移动端/小程序卡片列表                     │
│  ✓ 分步向导: 复杂检查项目通过向导引导用户逐步创建                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.3 设计决策汇总

| 编号 | 决策项 | 决策结果 |
|------|--------|---------|
| D01 | 年级在组织树中的定位 | 动态分组（年级作为班级属性，不作为实体节点） |
| D02 | 检查目标抽象 | 三类：ORGANIZATION、SPACE、USER |
| D03 | 检查项作用范围 | WHOLE（整体）/ INDIVIDUAL（关联个体） |
| D04 | 检查完整性 | 状态机：PENDING→IN_PROGRESS→COMPLETED/SKIPPED |
| D05 | 打分模式可比性 | 混合模式 + 等效映射或分开统计 |
| D06 | 模板与打分策略 | B+C结合：模板定义默认，项目可覆盖 |
| D07 | 嵌套检查打分 | 按检查层级配置不同模式 |
| D08 | 跳过权限 | 检查人员直接标记跳过 |
| D09 | 历史追溯 | 每条记录独立存储，按原始模式展示 |
| D10 | 最终分数计算 | 用户自定义公式 |
| D11 | 汇总周期处理 | 用户自定义（跳过天数处理可配置） |
| D12 | 并发检查 | 不允许并发，同一目标同一时间只能一人检查 |
| D13 | 加权系统 | 可选，按人数/容量加权 + 公平权重 |
| D14 | UI适配 | PC端双栏布局，移动端面包屑+卡片列表 |

### 2.2 三大检查主体

```
┌─────────────────────────────────────────────────────────────────────┐
│                      三大检查主体                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ORGANIZATION（组织）                                                │
│  ─────────────────────────────────────────────────────────────────  │
│  • 定义: 有层级结构的管理单元                                        │
│  • 特点: 可包含子组织、场所、用户                                    │
│  • 示例: 学校、系部、班级、部门、团队                                │
│  • 属性: 人数(member_count)、下属场所数(space_count)等               │
│  • 检查: 可检查组织整体，也可展开检查其包含的内容                     │
│                                                                     │
│  SPACE（场所）                                                       │
│  ─────────────────────────────────────────────────────────────────  │
│  • 定义: 物理或逻辑空间                                              │
│  • 特点: 归属于某个组织，可有使用者                                  │
│  • 示例: 宿舍、教室、办公室、会议室、车间                            │
│  • 属性: 容量(capacity)、面积(area)、当前使用人数(occupancy)等       │
│  • 检查: 检查场所本身，扣分可关联到具体使用者                        │
│                                                                     │
│  USER（用户）                                                        │
│  ─────────────────────────────────────────────────────────────────  │
│  • 定义: 系统中的人员                                                │
│  • 特点: 归属于某个组织，可使用某些场所                              │
│  • 示例: 学生、教师、员工、住户                                      │
│  • 属性: 角色(role)、职务(position)等                                │
│  • 检查: 检查个人表现，得分归集到所属组织                            │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.3 检查项的双重属性

每个检查项定义两个关键属性：

```yaml
检查项 (ScoreItem):
  # 基本信息
  name: "床铺不整洁"
  category: "卫生类"

  # 属性1: 适用于什么类型的目标
  applicable_to: [SPACE]        # 可选: ORGANIZATION, SPACE, USER

  # 属性2: 作用范围
  scope: INDIVIDUAL             # WHOLE=整体评价, INDIVIDUAL=需关联个体
  individual_type: USER         # 当scope=INDIVIDUAL时，关联的个体类型

  # 打分配置
  scoring:
    mode: DEDUCTION             # DEDUCTION/ADDITION/RATING/GRADE/CHECKLIST
    score: 2                    # 分值
    unit: "分/人"               # 单位描述
```

**作用范围说明**:

| scope | 说明 | 示例 |
|-------|------|------|
| WHOLE | 针对检查目标整体评价 | "班级整体纪律"、"宿舍整体卫生" |
| INDIVIDUAL | 需要关联到具体个体 | "学生迟到"、"床铺不整洁(关联学生)" |

### 2.4 打分模式

| 模式代码 | 名称 | 输入方式 | 输出格式 | 适用场景 |
|---------|------|---------|---------|---------|
| DEDUCTION | 扣分制 | 选择扣分项+数量 | 累计扣N分 | 宿舍卫生、纪律检查 |
| ADDITION | 加分制 | 选择加分项+数量 | 累计加N分 | 好人好事、特殊贡献 |
| BASE_SCORE | 百分制 | 基准分-扣分+加分 | 最终M分(满分100) | 综合评比 |
| RATING | 评分制 | 直接打分(1-N) | M/N分 | 教师评课 |
| GRADE | 评级制 | 选择等级 | A/B/C/D | 快速评估 |
| PASS_FAIL | 通过制 | 通过/不通过 | 通过/不通过 | 安全检查 |
| CHECKLIST | 清单制 | 逐项勾选 | M/N项(X%) | 合规检查 |

### 3.2 动态分组

```yaml
# 动态分组：基于属性筛选创建虚拟组
动态分组:
  id: 1
  name: "2024级全部班级"
  description: "按入学年份筛选的动态分组"

  # 筛选条件
  filter:
    entity_category: ORGANIZATION
    entity_type: CLASS
    conditions:
      - field: "attributes.enrollment_year"
        operator: "="
        value: 2024

  # 缓存（定期刷新）
  cached_entity_ids: [101, 102, 103, 104, ...]
  cached_at: "2026-02-01 00:00:00"

  # 自动刷新配置
  auto_refresh: true
  refresh_interval: "DAILY"  # HOURLY, DAILY, WEEKLY
```

## 六、加权与汇总

### 6.1 加权生效时机

```
检查执行时: 【不加权】原始得分
     │
     ▼
汇总到上级时: 【加权生效】
     │
     ├─ 场所 → 班级: 按容量/入住人数加权
     ├─ 班级 → 系部: 按班级人数加权
     └─ 用户 → 班级: 通常不加权
```

### 6.2 加权计算公式

```
              Σ (得分ᵢ × 权重ᵢ)
汇总得分 = ─────────────────────
                Σ 权重ᵢ
```

### 6.3 公平权重

解决检查次数不均导致的不公平问题:

| 模式 | 计算方式 | 效果 |
|------|---------|------|
| NONE | 不调整 | 直接平均 |
| NORMALIZE | 每次权重=1/检查次数 | 次数多少不影响均分 |
| BENCHMARK | 权重=基准次数/实际次数 | 鼓励接受更多检查 |

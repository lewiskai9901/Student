# 教务管理模块全面重构设计方案

> **日期**: 2026-03-29
> **版本**: v1.0
> **决策基础**: 学期固定课表+单双周 | 四级约束体系(全局/教师/班级/课程) | 合堂+走班双支持

---

## 一、现状问题诊断

### 1.1 架构问题（致命级）

| 问题 | 现状 | 影响 |
|------|------|------|
| 无领域模型 | 7个Controller直接操作JdbcTemplate | 业务逻辑散落在Controller中，无法复用、无法测试 |
| ID生成 | `System.currentTimeMillis()` | 并发冲突，数据覆盖 |
| 无事务管理 | 批量操作无`@Transactional` | 部分写入导致数据不一致 |
| SQL注入风险 | 字符串拼接WHERE条件 | 安全漏洞 |
| 硬编码用户 | `applicantId=1L` | 功能无法正常使用 |
| 无输入校验 | 日期范围、唯一性约束全靠数据库 | 错误信息不友好，数据质量差 |

### 1.2 排课系统问题（功能缺失级）

| 缺失功能 | 说明 |
|-----------|------|
| 班级开课管理 | 不知道每个班级本学期开哪些课、每门课几课时 |
| 课时分配 | 无法将课程分配给具体教师 |
| 教学班概念 | 只有行政班，不支持合堂/走班 |
| 约束引擎 | 无任何排课约束配置能力 |
| 自动排课算法 | API定义了但后端完全未实现 |
| 冲突检测 | 只有标记字段，无检测逻辑 |
| 教师偏好 | 表存在但无CRUD接口 |

### 1.3 需要删除/重构的文件

**删除（无用/空壳）：**
- 现有排课相关后端逻辑全部重写（Controller里的JdbcTemplate代码）
- `ScheduleView.vue` — 完全重构UI

**重构（保留数据模型但改造实现）：**
- 所有7个teaching Controller → 迁移到DDD架构
- `teaching.ts`（前端API）→ 按新架构重新组织
- `teaching.ts`（前端Types）→ 扩展约束/教学班类型

### 2.2 行政班 vs 教学班

这是整个排课系统的核心抽象：

```
行政班 (AdminClass)          教学班 (TeachingClass)
┌──────────────┐            ┌───────────────────┐
│ 一年级1班     │──────┐     │ 高数A班            │ ← 合堂：1班+2班一起上
│ 一年级2班     │──────┤     │ members: [1班,2班]  │
│ 一年级3班     │──┐   │     │ type: COMBINED     │
│              │  │   │     └───────────────────┘
│              │  │   │     ┌───────────────────┐
│              │  │   └────→│ 英语1班(基础)       │ ← 走班：跨班按能力分组
│              │  └────────→│ members: [1班部分,   │
│              │            │   2班部分, 3班部分]  │
└──────────────┘            │ type: WALKING      │
                            └───────────────────┘
                            ┌───────────────────┐
                            │ 一年级1班-语文      │ ← 普通：行政班=教学班
                            │ members: [1班]     │
                            │ type: NORMAL       │
                            └───────────────────┘
```

**教学班类型枚举：**
- `NORMAL` — 普通班（行政班 = 教学班，1:1）
- `COMBINED` — 合堂班（多个行政班合并，N:1）
- `WALKING` — 走班（跨行政班按能力/选科分层，M:N）

## 三、班级开课管理（新增模块）

### 3.1 业务流程

```
培养方案 ──→ 学期开课计划 ──→ 班级开课确认 ──→ 教学班组建 ──→ 教师分配 ──→ 排课
   │              │                │                │              │
   │              │                │                │              │
 定义课程体系   确定本学期      每个班开哪些课     合堂/走班分组    哪个老师教   生成课表
 (长期不变)    开哪些课        每门课几课时       组建教学班      分配工作量
```

### 3.2 数据模型：学期开课计划


## 四、排课约束引擎（核心新增）

### 4.1 四级约束体系

```
优先级从高到低:
┌────────────────────────────────────────────────────────┐
│ Level 1: 全局约束 (Global)                               │
│ 作用域: 整个学校/整个学期                                  │
│ 示例:                                                    │
│   - 每天最多排10节课                                       │
│   - 周二下午全校不排课（教研活动时间）                        │
│   - 第5节课为午休不排课                                     │
│   - 上午前两节优先排主课                                    │
├────────────────────────────────────────────────────────┤
│ Level 2: 教师约束 (Teacher)                               │
│ 作用域: 特定教师                                          │
│ 示例:                                                    │
│   - 张三只在周二、周四有空                                   │
│   - 李四每天最多3节课                                       │
│   - 王五的课不排在第1节（住得远）                             │
│   - 赵六周三下午固定开会不排课                                │
│   - 孙七要求课程尽量集中（不超过3天）                         │
├────────────────────────────────────────────────────────┤
│ Level 3: 班级约束 (Class)                                 │
│ 作用域: 特定行政班/教学班                                   │
│ 示例:                                                    │
│   - 一年级每天不超过6节课                                   │
│   - 毕业班下午不排新课                                      │
│   - 某班周五只排半天（社会实践）                              │
├────────────────────────────────────────────────────────┤
│ Level 4: 课程约束 (Course)                                │
│ 作用域: 特定课程                                          │
│ 示例:                                                    │
│   - 体育课只排下午                                         │
│   - 数学课排在上午前两节                                    │
│   - 实验课必须连排4节                                       │
│   - 同一门课每天最多排1次                                   │
│   - 英语课间隔排（不连续两天排）                             │
└────────────────────────────────────────────────────────┘
```

### 4.2 约束类型分类

```
约束按性质分为两类:

硬约束 (Hard Constraints) ── 必须满足，违反即无效
  ├── TIME_FORBIDDEN      时间禁排（指定时间段不能排课）
  ├── TIME_FIXED          时间固定（必须排在指定时间）
  ├── MAX_DAILY           每日上限（每天最多几节）
  ├── MAX_CONSECUTIVE     最大连排（连续排课不超过N节）
  ├── ROOM_REQUIRED       教室要求（必须在特定类型教室）
  └── NO_CONFLICT         不冲突（教师/教室/班级不能时间重叠）

软约束 (Soft Constraints) ── 尽量满足，有优先级权重
  ├── TIME_PREFERRED      时间偏好（优先排在某时段）
  ├── TIME_AVOIDED        时间回避（尽量不排在某时段）
  ├── SPREAD_EVEN         均匀分布（课程分散到不同天）
  ├── MORNING_PRIORITY    上午优先（主课尽量排上午）
  ├── COMPACT_SCHEDULE    紧凑排课（教师的课尽量集中）
  ├── MIN_GAP             最小间隔（同课程两次之间至少间隔N天）
  └── ROOM_PREFERRED      教室偏好（优先某间教室）
```

### 4.3 数据模型：约束配置

```sql
-- 排课约束规则表
CREATE TABLE scheduling_constraints (
    id BIGINT PRIMARY KEY,
    semester_id BIGINT NOT NULL COMMENT '学期ID',
    constraint_name VARCHAR(100) NOT NULL COMMENT '约束名称（人类可读）',

    -- 约束级别与目标
    constraint_level TINYINT NOT NULL COMMENT '1全局 2教师 3班级 4课程',
    target_id BIGINT COMMENT '目标ID（教师ID/班级ID/课程ID，全局时为空）',
    target_name VARCHAR(100) COMMENT '目标名称（冗余，方便显示）',

    -- 约束类型与性质
    constraint_type VARCHAR(50) NOT NULL COMMENT '约束类型枚举',
    is_hard TINYINT DEFAULT 1 COMMENT '1硬约束 0软约束',
    priority INT DEFAULT 50 COMMENT '优先级权重（软约束用，1-100，越大越重要）',

    -- 约束参数（JSON，不同类型不同结构）
    params JSON NOT NULL COMMENT '约束参数',
    /*
    params 示例:

    TIME_FORBIDDEN:
      {"days": [2], "periods": [5,6,7,8], "reason": "周二下午教研"}
      {"days": [1,2,3,4,5], "periods": [5], "reason": "午休"}

    TIME_FIXED:
      {"day": 3, "periods": [1,2], "reason": "固定周三上午1-2节"}

    TIME_PREFERRED:
      {"days": [1,3,5], "periods": [1,2,3,4], "reason": "优先周一三五上午"}

    MAX_DAILY:
      {"maxPeriods": 6, "maxSessions": 3}

    MAX_CONSECUTIVE:
      {"maxPeriods": 2}

    SPREAD_EVEN:
      {"minGapDays": 1, "reason": "至少间隔1天"}
```

## 五、冲突检测系统

### 5.1 冲突类型

```
冲突类型分类:

1. 资源冲突（硬冲突，必须解决）
   ├── TEACHER_CONFLICT    — 同一教师同一时间被排了两节课
   ├── CLASSROOM_CONFLICT  — 同一教室同一时间被排了两个班
   ├── CLASS_CONFLICT      — 同一班级同一时间有两门课
   └── STUDENT_CONFLICT    — 走班模式下同一学生同一时间有两门课

2. 约束冲突（预检测，排课前发现）
   ├── CONSTRAINT_MUTUAL   — 两个约束互相矛盾（如A要求只排周二，B要求周二禁排）
   ├── CONSTRAINT_INFEASIBLE — 约束导致无法排完所有课（可用时间段不够）
   └── CAPACITY_OVERFLOW   — 教室容量不够

3. 软冲突（警告，不阻塞）
   ├── PREFERENCE_VIOLATED — 教师偏好未满足
   ├── SPREAD_UNEVEN       — 课程分布不均匀
   └── OVERLOAD_WARNING    — 某天课程过于集中
```

## 十三、技术要点

### 13.1 自动排课算法选型

| 算法 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 遗传算法(GA) | 全局搜索能力强，处理多目标优化 | 收敛慢，参数敏感 | 大规模排课 |
| 约束满足(CSP) | 精确满足硬约束，可证明无解 | 规模大时慢 | 约束严格场景 |
| 模拟退火(SA) | 实现简单，跳出局部最优 | 结果不稳定 | 小规模快速排课 |
| **推荐: GA + CSP混合** | 兼顾全局与精确 | 实现复杂 | **本项目** |

**推荐方案**：CSP生成可行初始解 → GA优化软约束满足度

### 13.2 性能考虑

- 排课计算应异步执行（后端用 `@Async` 或消息队列）
- 前端WebSocket推送排课进度
- 课表查询按学期+维度索引，避免全表扫描
- 约束检查结果缓存（约束不变则不重复计算）

### 13.3 数据一致性

- 所有排课操作需要事务保护
- 发布课表后锁定，只能通过调课流程修改
- 使用乐观锁防止并发编辑冲突
- 约束变更后自动触发冲突重检
